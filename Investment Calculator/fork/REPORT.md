## 2. Контрольні питання та відповіді

### Питання 1: Які основні функції виконує система Maven у процесі життєвого циклу програмного проєкту на Java?

**Відповідь:**
Maven виконує наступні основні функції:
- **Управління залежностями** - автоматичне завантаження необхідних бібліотек з репозиторіїв
- **Стандартизація структури проєкту** - забезпечує єдину структуру каталогів для всіх проєктів
- **Автоматизація збірки** - компіляція, тестування, пакування в єдиному процесі
- **Управління життєвим циклом** - визначає фази від ініціалізації до розгортання
- **Генерація документації** - автоматичне створення документації проєкту
- **Управління версіями** - контроль версій артефактів та залежностей
- **Інтеграція з CI/CD** - легка інтеграція з системами безперервної інтеграції

### Питання 2: У чому полягає відмінність між декларативною моделлю проєкту Maven і традиційним сценарним підходом до збирання програмних продуктів?

**Відповідь:**
Основні відмінності:

**Декларативна модель Maven:**
- Описує ЩО потрібно зробити, а не ЯК
- Використовує стандартні конвенції та структуру
- Конфігурація через XML (pom.xml)
- Автоматичне виконання стандартних завдань
- Менше коду конфігурації

**Традиційний сценарний підхід (Ant, Make):**
- Описує покроково ЯК виконувати завдання
- Потребує детального опису кожного кроку
- Більше контролю, але більше коду
- Немає стандартної структури проєкту
- Складніше підтримувати та переносити між проєктами

### Питання 3: Яке призначення має файл pom.xml і які ключові елементи він містить для опису конфігурації проєкту?

**Відповідь:**
Файл pom.xml (Project Object Model) - це центральний конфігураційний файл Maven-проєкту.

**Ключові елементи:**
- `<groupId>` - ідентифікатор організації/групи
- `<artifactId>` - унікальний ідентифікатор проєкту
- `<version>` - версія проєкту
- `<packaging>` - тип пакування (jar, war, pom)
- `<name>` - ім'я проєкту
- `<dependencies>` - список залежностей
- `<properties>` - властивості проєкту
- `<build>` - конфігурація збірки
- `<plugins>` - плагіни для розширення функціональності

### Питання 4: Як організовано стандартну структуру каталогів Maven і яке значення вона має для узгодженості процесу розробки?

**Відповідь:**
**Стандартна структура Maven:**
```
project/
├── pom.xml
├── src/
│   ├── main/
│   │   ├── java/       - вихідний код Java
│   │   └── resources/  - ресурси програми
│   └── test/
│       ├── java/       - тестовий код
│       └── resources/  - тестові ресурси
└── target/             - результати збірки
```

**Значення:**
- Уніфікація між проєктами
- Легше розуміння чужого коду
- Автоматична інтеграція з IDE
- Швидша адаптація нових розробників
- Сумісність інструментів
- Зменшення помилок конфігурації

### Питання 5: Що таке артефакт у Maven і як визначаються його основні атрибути у конфігураційному файлі?

**Відповідь:**
**Артефакт** - це результат збірки Maven-проєкту (JAR, WAR, POM файл).

**Основні атрибути (координати Maven):**
- `groupId` - ідентифікатор групи/організації (com.investmentcalc)
- `artifactId` - унікальне ім'я артефакту (investment-calculator)
- `version` - версія артефакту (1.0.0)
- `packaging` - тип пакування (jar)
- `classifier` - додатковий класифікатор (опціонально)

Ці атрибути однозначно ідентифікують артефакт у репозиторії.

### Питання 6: Яке призначення мають репозиторії Maven і яку роль відіграють локальний та центральний репозиторії у процесі збирання?

**Відповідь:**
**Типи репозиторіїв:**

**Локальний репозиторій (~/.m2/repository):**
- Кеш завантажених залежностей
- Зберігає локально встановлені артефакти
- Прискорює повторні збірки
- Зменшує мережевий трафік

**Центральний репозиторій (Maven Central):**
- Головне сховище публічних бібліотек
- Містить тисячі open-source проєктів
- Автоматично доступний всім проєктам
- Джерело офіційних версій бібліотек

**Корпоративний репозиторій (Nexus, Artifactory):**
- Приватні артефакти компанії
- Проксі для центрального репозиторію
- Контроль версій внутрішніх бібліотек

### Питання 7: Як за допомогою плагінів Maven здійснюється створення виконуваного JAR-файла прикладного застосунку?

**Відповідь:**
Використовуються наступні плагіни:

**maven-jar-plugin:**
```xml
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-jar-plugin</artifactId>
    <configuration>
        <archive>
            <manifest>
                <mainClass>com.investmentcalc.InvestmentCalculator</mainClass>
            </manifest>
        </archive>
    </configuration>
</plugin>
```

**maven-shade-plugin (для fat-jar):**
```xml
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-shade-plugin</artifactId>
    <executions>
        <execution>
            <phase>package</phase>
            <goals>
                <goal>shade</goal>
            </goals>
        </execution>
    </executions>
</plugin>
```

Команда `mvn package` створює JAR-файл у каталозі target/.

### Питання 8: У чому полягає відмінність між звичайним JAR і архівом типу fat-jar створеним за допомогою плагіна maven-shade-plugin?

**Відповідь:**

**Звичайний JAR:**
- Містить лише код проєкту
- Не включає залежності
- Менший розмір файлу
- Потребує classpath з усіма залежностями
- Запуск: `java -cp lib/*:app.jar MainClass`

**Fat-jar (uber-jar):**
- Включає всі залежності всередині
- Самодостатній виконуваний файл
- Більший розмір
- Не потребує зовнішніх бібліотек
- Запуск: `java -jar app.jar`
- Легше розповсюдження
- Ідеальний для десктоп-застосунків

### Питання 9: Як Maven забезпечує відтворюваність і незалежність процесу збирання від конкретного середовища розробки?

**Відповідь:**
Maven забезпечує відтворюваність через:

1. **Декларативна конфігурація** - всі налаштування в pom.xml
2. **Стандартна структура** - однакова для всіх проєктів
3. **Версіонування залежностей** - точні версії бібліотек
4. **Центральні репозиторії** - єдине джерело артефактів
5. **Стандартні фази збірки** - однаковий процес для всіх
6. **Профілі** - адаптація під різні середовища
7. **Властивості** - параметризація конфігурації
8. **Плагіни з версіями** - фіксовані версії інструментів

Результат: однаковий результат збірки на будь-якій системі.
